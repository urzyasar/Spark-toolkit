Per-Executor JVM Memory Layout i
How this is computed
Step 1: spark.executor.memory = execM GB (heap)
Step 2: Managed pool = heap × spark.memory.fraction
  = execM × mf
Step 3: Storage = managed × spark.memory.storageFraction
  = managed × sf
Step 4: Execution = managed × (1 − storageFraction)
  = managed × (1 − sf)
Step 5: User = heap × (1 − memFraction)
  = execM × (1 − mf)
Step 6: Overhead = max(384MB, 10% × heap) off-heap
  PySpark adds +1024MB to overhead
✦ Execution + Storage share the managed pool. If execution needs more, it borrows from storage (not from user).

Execution = shuffle, aggregation, joins (spills to disk if exceeded). Storage = cached RDD/DF data (evicted if execution needs more). User = your UDFs, data structures. Overhead = JVM metaspace + off-heap (not in heap budget). PySpark OH = Python worker process (shown when PySpark toggle ON). Click ⓘ for full computation steps.

③ 🚀 Deploy Modes Explained
CLIENT Mode: Driver runs on the submitting machine (master node / laptop / edge node). This node is NOT a worker — its resources do NOT count toward cluster executor allocation. However, the master node's memory and CPU are consumed by the driver process. The tool shows this as a separate "Driver on Master" indicator with risk assessment for master node overuse.

CLUSTER Mode: Driver is deployed on a worker node inside the cluster. One worker node is reserved for the driver. That node's cores and memory are consumed by the driver and are NOT available for executors. The tool deducts this from available resources automatically.

STANDALONE Mode: Spark's own resource manager (no YARN/K8s). The master is a separate coordinator process — it does NOT run executors. Workers register with master. Driver can be client or cluster mode within standalone.
✦ For production YARN: use CLUSTER mode so the driver is managed by YARN and tolerates driver failures via restart policies.
④ 📊 Visualization Tab — What Each Chart Means
Memory Doughnut — full cluster RAM broken into: Executor Heap, Off-heap Overhead, Driver (cluster mode only), OS+RM Reserves, Unused. Target: Executor Heap 60–75% of total.

Core Bar — physical cores split by use. Unused cores = headroom. HDFS needs ≤5 cores/executor for max I/O throughput.

Utilization Bars — used vs available in absolute numbers. Target: 50–80% for both CPU and RAM. Note: % is calculated against usable capacity (OS+RM reserves excluded from denominator) — this reflects true executor pressure, not total node RAM.

Per-Executor JVM Layout — shows Execution memory, Storage memory, User memory, JVM overhead, and PySpark overhead separately (when PySpark toggle is ON). Click the ⓘ icon on the chart title to see the full step-by-step computation formula.

⑤ ⚡ Parallel Jobs — Full Workflow
Step 1 — Pre-Allocated Jobs (fixed reservations): Use for always-on jobs — streaming pipelines, ML inference, scheduled ETL with SLAs. Set executor count, cores/exec, memory/exec directly. These are deducted from the total pool first.

Step 2 — Remaining Pool = 100%: After pre-alloc, the leftover is the 100% baseline. The colored pool bar shows exactly what's used and what's available.

Step 3 — Concurrent Jobs %: Each job's slider sets its share of the remaining pool. Click ⚡ Auto-compute to get recommended minimums based on pool size.

Step 4 — Min Executors per Job: Guarantees each job gets enough JVM processes. Below 2 = no fault tolerance. Toggle ON/OFF to enforce or just warn.

Step 5 — Min Parallel Tasks per Job: = min executors × cores/executor. Fewer tasks than partitions = task queuing and idle cores.

PySpark Overhead: When the PySpark toggle is ON in the Configuration tab, the +1024MB Python worker overhead is automatically included in every executor's spark.executor.memoryOverhead for all parallel jobs. The pool bar "Overhead" column and the per-job charts both include this. The Computation Trace (Step 4 per job) shows the breakdown explicitly.

Memory per Executor: Each parallel job receives the global spark.executor.memory value by default. Use the "Memory per Core" override to change memory specifically for parallel jobs (e.g. 8GB/core for ML jobs that need more heap).
✦ If no pre-allocated jobs: 100% of all cluster executor resources flow directly to the parallel job planner.
⑥ 🔑 Concurrent Tasks vs Task Slots — The Key Concept
Concurrent Tasks = the maximum number of Spark tasks that can run simultaneously across your cluster.

Formula: Concurrent Tasks = Total Executors × Cores per Executor
Example: 20 executors × 5 cores = 100 tasks running in parallel at once

This is NOT:
• The total number of tasks in your job (that depends on partitions)
• The number of partitions (set that via spark.default.parallelism)
• The degree of data parallelism (set via shuffle partitions)

Why it matters for sizing:
• If concurrent tasks < partition count → tasks queue, some cores sit idle
• If concurrent tasks >> partition count → many executors sit idle after first wave
• Ideal: partition count = 2–3× concurrent tasks (keeps cores busy with wave overlap)

spark.default.parallelism = default partition count for RDD ops (set to 2–3× concurrent tasks)
spark.sql.shuffle.partitions = partition count after DataFrame shuffles (same recommendation)
✦ AQE (Spark 3.x) auto-coalesces small shuffle partitions — set shuffle.partitions higher and let AQE reduce them at runtime.
⑦ 📋 Tuning Loop — Observe → Adjust → Regenerate
Apply config → run in Spark UI → identify bottleneck:

GC time >10%: Executor heap too large or too many cores per executor. Reduce executor memory or executor cores.
Tasks spilling to disk: Execution memory insufficient. Increase executor memory, or raise spark.memory.fraction.
Many tiny tasks: Too many partitions. Reduce parallelism or enable AQE coalescing.
Data skew: Enable AQE (skewJoin.enabled=true), or repartition by a better key before joins.
Executors idle between stages: Partition count too low. Concurrent tasks > partitions → some executors wait.
Shuffle read/write very large: Join/groupBy before filter — reorder operations, or increase executor memory.
OOM on executor: Check memoryOverhead — may need +512MB to +2GB for Python/native code.
✦ Spark UI → Stages → Task Time Distribution: if GC time is colored red, executor memory is the bottleneck.
Quick Reference Formulas
Executor Sizing
exec_cores = min(5, usable/node)
exec/node = floor(usable ÷ cores)
exec_mem = availRAM/exec × 0.875
overhead = max(384, 10% × heap)
Parallelism
total_exec = nodes × exec/node
concur_tasks = total_exec × cores
parallelism = concur_tasks × 2
shuffle_pts = parallelism (AQE+)
Thresholds
